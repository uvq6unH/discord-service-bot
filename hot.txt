## Code Review — discord-service-bot-render

---

### 🏗️ Kiến trúc & Cấu trúc

**`bot.js` quá lớn và làm quá nhiều việc (2240 dòng)**

Đây là vấn đề lớn nhất. File này chứa: game logic (blackjack, poker, slots...), command handlers (30+ commands), interaction handlers, keepalive, utility functions — tất cả trong một file. Hậu quả:
- Khó debug khi có lỗi
- Không thể test từng phần độc lập
- Mỗi lần sửa một game lại phải mở file 2000 dòng

Cần tách thành ít nhất: `src/games/blackjack.js`, `src/games/poker.js`, `src/commands/`, `src/interactions/`.

**`blackjackSessions` và `pokerSessions` là global `Map` trong module**

```js
const blackjackSessions = new Map();
const pokerSessions = new Map();
```

Những map này không bao giờ bị GC — session hết hạn được xóa bằng `setTimeout`, nhưng nếu process restart giữa chừng (Render free tier restart rất thường xuyên), map mất sạch. Code đã có Redis fallback qua `stateStore` nhưng việc dùng song song cả hai backend (in-memory Map + Redis) tạo ra inconsistency (xem bug bên dưới).

**`lolAccounts` và TFT dùng chung key Redis của LoL**

TFT link account đang lưu vào `guild.lolAccounts` — không có `tftAccounts` riêng. Nếu user link LoL và TFT với 2 account khác nhau thì conflict.

### 🐛 Bug tiềm ẩn & Độ ổn định

**Race condition trong blackjack multi-player**

Khi 2 người bấm nút cùng lúc (rất dễ xảy ra):

```js
// Cả hai đều vào đây đồng thời
const session = await getGameSession(client, 'blackjack', blackjackSessions, ...);
// ... modify session ...
await persistGameSession(client, 'blackjack', messageId, session);
```

Không có lock hay optimistic concurrency — người thứ 2 overwrite kết quả của người thứ 1. Đặc biệt nguy hiểm ở `adjustBalance` + `persistGameSession` vì đây là hai operation riêng biệt, không atomic.

**Session double-refund khi restart**

Khi Render restart process:
1. `purgeStaleGameSessions()` chạy lúc boot → refund bets từ Redis
2. Nhưng nếu session đã được `expireSessionWithRefund()` trước khi restart (chạy trong memory) mà chưa kịp xóa khỏi Redis → refund lần 2

Nguyên nhân: `expireSessionWithRefund` refund tiền trực tiếp qua `adjustBalance` nhưng sau đó mới gọi `deleteGameSession` — nếu crash giữa 2 lệnh này, lần boot sau `purgeStaleGameSessions` thấy session vẫn còn và refund lại.

**`getGameSession` revive session từ Redis không set `createdAt`**

```js
session = await client.stateStore.getGameSession(guildId, type, messageId);
if (type === 'poker') revivePokerSession(session);
if (session?.status === 'active') {
  session.timeout = scheduleSessionExpiry(...); // TTL từ bây giờ, không phải từ createdAt
```

Session revive từ Redis sẽ được cấp thêm 2 tiếng từ thời điểm revive, không phải từ lúc tạo. Session có thể tồn tại > 4 tiếng.

**`shouldDealerHit` dùng `Math.random()` — dealer không deterministic**

```js
if (dealerValue === 17) return chance(25); // 25% hit on hard 17
```

Dealer blackjack chuẩn phải stand trên hard 17. Đây có thể là chủ ý (casino edge), nhưng nếu không thì đây là bug logic nghiêm trọng — player có thể bị thua bởi dealer hit khi không nên.

**`purge` command không xử lý messages > 14 ngày**

`bulkDelete` của Discord không xóa được message cũ hơn 14 ngày — nó sẽ throw error nhưng code chỉ báo "Deleted X messages" mà không báo lỗi cho user.

**`mergeWithDefaultCommands` trong configStore có thể tạo duplicate**

```js
for (const command of defaultConfig.commands) {
  if (!existingTypes.has(command.type) && !existingNames.has(command.name)) {
    merged.push(command);
  }
}
```

Check theo cả `type` lẫn `name` — nếu user đổi tên lệnh `lolprofile` thành `lol2` (giữ `type: 'lolprofile'`) thì default vẫn được merge thêm vào, kết quả có 2 lệnh cùng type.

### 🔒 Bảo mật

**Rate limiter tự viết có memory leak**

```js
const hits = new Map(); // Không bao giờ cleanup entries cũ
```

Map này chứa tất cả IPs đã request, không có cleanup job. Sau vài ngày uptime, đây là memory leak. Nếu có nhiều IPs khác nhau (bot scan), map tăng không giới hạn.

Fix: dùng `setInterval` để xóa entries đã expired, hoặc dùng `express-rate-limit`.

**`/api/status` không cần auth — leak thông tin**

```js
app.get('/api/status', async (_req, res) => {
  res.json({ botReady, botUser, guildCount, configuredGuilds });
});
```

Endpoint này public, trả về `botUser` (tag của bot), số guild, số config. Không critical nhưng nên require auth.

**OAuth callback lộ error message trực tiếp ra client**

```js
res.status(500).send(`Login failed: ${err.message}`);
```

`err.message` có thể chứa internal details (Discord API response, URL...). Nên log server-side và trả về generic message.

**`announcementMention` không validate**

```js
announcementMention: String(patch.announcementMention ?? '').trim().slice(0, 100),
```

Field này được đưa thẳng vào message Discord: `${config.announcementMention}\n${announcement}`. Nếu user nhập `@everyone` hoặc `@here`, bot sẽ ping toàn server. Nên kiểm tra bot có permission `Mention Everyone` trước khi gửi.

**`say` command xóa message gốc nhưng không kiểm tra bot có permission xóa**

```js
await source.delete().catch(() => null); // silent fail
return channel.send(messageText);
```

Nếu bot không có quyền xóa, message gốc vẫn còn nhưng bot vẫn gửi → user thấy cả hai. Nên báo lỗi thay vì silent fail.

### ⚡ Performance & Redis

**`getGuildConfig` gọi nhiều lần per-request không cache**

Mỗi interaction/message event gọi `configStore.getGuildConfig(guildId)` — function này `clone()` toàn bộ defaultConfig mỗi lần (deep JSON parse/stringify). Với server active, đây là nhiều clone không cần thiết.

**Upstash `_redisSaveGuild` không dùng pipeline**

```js
async _redisSaveGuild(guildId, guild) {
  await this._redis._request(['SADD', this._guildIndexKey(), guildId]); // request 1
  await this._redis.set(this._guildKey(guildId), guild);                 // request 2
}
```

Mỗi lần save = 2 round-trips đến Upstash. Dùng pipeline thì 1 round-trip. Với game session update thường xuyên, đây là overhead đáng kể.

**`/api/guilds` fetch members cho toàn bộ guilds đồng thời**

```js
const manageableGuilds = await Promise.all([...botClient.guilds.cache].map(async ([id, guild]) => {
  const member = await guild.members.fetch(userId).catch(() => null);
```

Nếu bot ở 50 servers, endpoint này gửi 50 Discord API calls song song. Với Render free tier, điều này có thể trigger Discord rate limit.

**DDragon cache không có size limit**

`Cache` class trong `lolApi.js` không giới hạn số entries. Nếu có nhiều người query nhiều champions/items khác nhau, cache tăng mãi. Với Render free 512MB RAM, đây là rủi ro.

**`@discordjs/voice`, `ffmpeg-static`, `play-dl` trong dependencies nhưng không dùng**

`package.json` có 3 packages voice/audio nặng (~50MB) nhưng không có code nào dùng chúng. Tăng install time, tăng memory footprint, tăng attack surface.

### Tóm tắt ưu tiên sửa

**Cần sửa ngay (ảnh hưởng đến tính chính xác của dữ liệu):**
1. Race condition blackjack multi-player → thêm mutex/lock per messageId
2. Session double-refund khi restart → mark session "refunded" trước khi adjustBalance
3. Memory leak trong rate limiter → thêm cleanup

**Nên sửa sớm:**
4. Tách `bot.js` thành modules
5. `_redisSaveGuild` dùng pipeline
6. Xóa 3 packages voice không dùng
7. Rate limit `/api/guilds`

**Nice to have:**
8. Auth cho `/api/status`
9. Generic error message trong OAuth callback
10. Validate `announcementMention`
11. Size-bound cache cho DDragon
